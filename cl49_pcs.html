How the RS layer hands 10 Gbps of Ethernet traffic to the PCS over the XGMII bus, and how the PCS encodes it into 66-bit blocks for the line.

Where PCS sits in the sublayer stack

PCS is not adjacent to the MAC directly — the Reconciliation Sublayer (RS) sits between them and presents data via XGMII. On the other side, FEC (when present) sits between PCS and the PMA.

02

XGMII bus structure

XGMII is a full-duplex parallel bus — TX and RX paths are completely independent and unidirectional. Each direction carries the same set of signals.

Signal Width Direction Description
TXD[31:0] 32 bits RS → PCS 4 data octets per cycle — one octet per lane (lane 0 = bits 7:0, lane 3 = bits 31:24)
TXC[3:0] 4 bits RS → PCS One control flag per lane. 1 = the byte on that lane is a control code, not data
TX_CLK 1 bit RS → PCS 312.5 MHz single data rate (SDR) clock. Data captured on rising edge
RXD[31:0] 32 bits PCS → RS 4 received octets per cycle from the PCS decode path
RXC[3:0] 4 bits PCS → RS Control flags for each received lane — mirrors TXC semantics
RX_CLK 1 bit PCS → RS Recovered RX clock — derived from the incoming serial stream by the PMA CDR
★

TX_CLK and RX_CLK are independent. TX clock is generated locally; RX clock is recovered from the incoming bitstream. They may differ slightly in frequency due to spread-spectrum clocking on the remote transmitter.

The actual line rate after 64B/66B encoding is 10.3125 Gbps — 10 Gbps × (66÷64) = 3.125% overhead. The PMA runs at 10.3125 Gbps, not 10 Gbps.

ℹ

XGMII is an on-chip interface only. It is not brought out to PCB connectors. For chip-to-chip 10G connections, XAUI (4 × 3.125 Gbps serial lanes, Clause 48) is used instead.

04

XGMII control codes

When a TXC bit is 1, the corresponding lane carries one of these four special values instead of payload data. The PCS must recognise all four and translate them into the appropriate 64B/66B control block.

0x07
/I/ IDLE
Sent continuously between frames as inter-frame gap. All 4 lanes carry 0x07 with TXC=1111 during idle periods.
TXC = 1 on any lane
0xFB
/S/ START
Marks the first byte of a new Ethernet frame. Must always appear on lane 0 only. Lanes 1–3 in the same cycle carry the first 3 bytes of preamble.
TXC[0] = 1, lane 0 only
0xFD
/T/ TERMINATE
Marks the last valid byte of a frame. Can appear on any lane (0–3). Lanes after /T/ in the same cycle are filled with IDLE. Lane position encodes how many valid bytes precede it.
TXC[n] = 1, any lane
0xFE
/E/ ERROR
Propagates an error condition from the MAC. PCS encodes this as an error control block so the remote receiver is notified.
TXC[n] = 1, any lane
★

The START code rule is strict: if the RS cannot place /S/ on lane 0 due to frame alignment, it must insert extra IDLE cycles until alignment is correct. This guarantees the PCS always finds /S/ at a predictable position.

The TERMINATE lane position matters to the PCS: it determines how many valid data bytes are in that cycle, which determines which 64B/66B control block type (O-code) gets generated. There are 8 possible terminate positions (lane 0–3 in two possible alignment phases).

How PCS converts XGMII → 64B/66B blocks

The PCS accumulates two consecutive XGMII cycles (8 bytes) and encodes them into one 66-bit block. The 2-bit sync header identifies whether the block carries data or control information.

ℹ

After 64B/66B encoding, the 66-bit block passes through the scrambler (polynomial x⁵⁸ + x³⁹ + 1). The scrambler randomises the payload bits to prevent long runs of identical bits that would break clock recovery at the receiver. The 2-bit sync header is not scrambled.
